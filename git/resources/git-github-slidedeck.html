Setting Up Your Identity

Why is this needed?
Every commit you make is stamped with your name and email ‚Äî so your team knows who made each change.

Step 3 of 8
Branching ‚Äî Experiment Safely
Work on risky features without breaking the main code
üç≥ Restaurant Kitchen Analogy:
The main branch is the restaurant's main kitchen. A new branch is a separate "taste kitchen" where you safely experiment with new recipes without disrupting service.
  • A branch is an identical copy of your code
  • Changes on a branch don't affect main
  • Work freely ‚Äî commit as much as you want
  • Merge back when the feature is complete
# Create a new branch git branch dark-mode # Switch to the new branch git checkout dark-mode # Shortcut: create AND switch git checkout -b dark-mode # See all branches (* = current) git branch # Make changes, then commit normally git add . git commit -m "Added dark mode UI"

Step 5 of 8
Merging ‚Äî Combine Your Work
Bring completed feature branches back into the main codebase
üß© Puzzle Analogy:
Merging is like combining completed puzzle pieces (feature branch) back into the main picture (main branch) ‚Äî the result is the complete, final image.
  • Always switch to the target branch first (usually main)
  • Then merge the source branch into it
  • Git automatically combines non-conflicting changes
  • If same lines changed ‚Äî Git signals a merge conflict to resolve manually

Step 6 of 8
Fixing Mistakes
Git gives you total control to undo and recover
Before Commit

git restore <filename>

Accidentally modified a file but haven't committed yet? This instantly reverts the file to its last committed (safe) state. Like pressing Ctrl+Z back to the last save point.

After Commit

git revert <commit-id>

Realized an older commit introduced a bug? Creates a NEW commit that undoes it ‚Äî keeping history clean and traceable. Safe to use on shared branches.

# Scenario 1: Oops! Wrong edit, not committed git restore index.html # File is back to last committed state ‚úì # Scenario 2: Revert a committed mistake # First, find the bad commit ID git log --oneline # abc1234 Added dark mode (this broke things!) # def5678 Fixed the bug # Revert the bad commit safely git revert abc1234 # Creates new commit undoing abc1234 ‚úì # DANGER ZONE: Hard reset (rewrites history) git reset --hard HEAD~1 # ‚ö†Ô∏è Only use on local branches!
‚ö†Ô∏è Key Rule: Use git revert for shared/public branches (safe, adds new commit). Use git reset only on your own private branches (destructive, rewrites history).

Step 8 of 8
Pull Requests ‚Äî Code Review
The professional way to merge code in a team
üëÆ Quality Control Analogy:
A Pull Request (PR) is like submitting your work to a quality inspector before it goes to production. Your team reviews every line before it reaches the main codebase.
  • Rarely merge your own branches in real teams
  • Push your branch to GitHub, then open a PR
  • PR shows exactly which lines were added or removed
  • Team reviews, comments, requests changes
  • Once approved, click "Merge pull request"
# 1. Push your feature branch git push origin dark-mode # 2. Go to GitHub.com ‚Üí your repo # Click "Compare & pull request" # Add a title and description # Click "Create pull request" # 3. Team reviews your code on GitHub # They may request changes # Make changes locally, commit & push git add . git commit -m "Applied review feedback" git push origin dark-mode # 4. Once approved ‚Üí Merge on GitHub! # Main branch stays stable ‚úì
